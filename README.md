## Overview

This project automates **CIS Ubuntu 22.04 LTS Level 1 Benchmark** hardening on AWS EC2 instances using Ansible, with Terraform for infrastructure provisioning and automated compliance reporting in plain text and HTML.

| CIS Section | Coverage |
|---|---|
| 1 — Filesystem Configuration | Disable unused kernel filesystem modules, configure login banners |
| 2 — Services | Stop and disable unnecessary services, remove insecure client packages |
| 3 — Network | sysctl hardening: IP forwarding, ICMP redirects, SYN cookies, reverse path filtering |
| 4 — Logging & Auditing | auditd installation and audit rules, rsyslog installation and file permissions |
| 5 — Access Control | SSH daemon hardening, password policy, PAM password quality |

## Enterprise Context

In a production **Ansible Automation Platform (AAP) / AWX** environment, this project maps directly to:

- **Job Templates** — `site.yml` and `audit.yml` become separate Job Templates; `audit.yml` is scheduled nightly as a read-only compliance check
- **Surveys** — AAP Surveys expose `managed_node_count` and `disable_ipv6` as runtime overrides without editing playbooks
- **Credentials** — SSH private keys are stored in AAP Credential objects, never on disk
- **Notifications** — Failed jobs trigger PagerDuty or Slack alerts via AAP Notification Templates
- **Inventories** — The `inventory/hosts.ini` generated by Terraform would be replaced by a dynamic AWS inventory plugin in AAP
- **RBAC** — The `cis_hardening` role can be published to a private Automation Hub for reuse across teams

## Step-by-Step Execution

### Step 1 — Clone the repository

```bash
git clone https://github.com/sabatino/ansible-linux-hardening.git
cd ansible-linux-hardening
```

### Step 2 — Generate SSH key pair

```bash
ssh-keygen -t rsa -b 4096 -f ~/.ssh/cis_hardening -N ""
```

### Step 3 — Provision infrastructure with Terraform

```bash
cd terraform
terraform init
terraform plan \
  -var="public_key_path=~/.ssh/cis_hardening.pub" \
  -var="private_key_path=~/.ssh/cis_hardening"

terraform apply \
  -var="public_key_path=~/.ssh/cis_hardening.pub" \
  -var="private_key_path=~/.ssh/cis_hardening" \
  -auto-approve
cd ..
```

Expected output (abbreviated):

```
Apply complete! Resources: 7 added, 0 changed, 0 destroyed.

Outputs:

control_node_public_ip     = "54.211.x.x"
managed_nodes_public_ips   = ["54.211.y.y", "54.211.z.z"]
ssh_control_node           = "ssh -i ~/.ssh/cis_hardening ubuntu@54.211.x.x"
monthly_cost_estimate      = "t3.micro: ~$8.47/month ..."
```

Terraform also writes `inventory/hosts.ini` automatically.

### Step 4 — Verify Ansible connectivity

```bash
ansible all -i inventory/hosts.ini -m ping
```

Expected output:

```yaml
managed-node-01 | SUCCESS => {
    "ping": "pong"
}
managed-node-02 | SUCCESS => {
    "ping": "pong"
}
```

### Step 5 — Run read-only audit BEFORE hardening

```bash
ansible-playbook -i inventory/hosts.ini playbooks/audit.yml
```

This establishes the baseline compliance score before any changes.

### Step 6 — Run full hardening

```bash
ansible-playbook -i inventory/hosts.ini playbooks/site.yml
```

## Selective Execution with Tags

| Tag | Controls Applied |
|---|---|
| `filesystem` | CIS Section 1 — Filesystem modules and login banners |
| `services` | CIS Section 2 — Service disabling and package removal |
| `network` | CIS Section 3 — sysctl kernel parameters |
| `logging` | CIS Section 4 — auditd and rsyslog |
| `access` | CIS Section 5 — SSH and password policy |
| `report` | Section 6 — Report generation only (always runs) |
| `cis_1` … `cis_5` | Alternative numeric tag per section |

```bash
# Apply network hardening only
ansible-playbook -i inventory/hosts.ini playbooks/site.yml --tags network

# Dry run — show what would change without applying
ansible-playbook -i inventory/hosts.ini playbooks/site.yml --check

# Dry run for a specific section
ansible-playbook -i inventory/hosts.ini playbooks/site.yml --tags access --check
```

## Customization

Override any default variable at runtime with `-e`:

```bash
# Use a custom SSH port
ansible-playbook -i inventory/hosts.ini playbooks/site.yml \
  -e "ssh_port=2222"

# Keep IPv6 disabled
ansible-playbook -i inventory/hosts.ini playbooks/site.yml \
  -e "disable_ipv6=true"

# Relax password max age for a specific environment
ansible-playbook -i inventory/hosts.ini playbooks/site.yml \
  -e "password_max_days=90"
```

Per-environment variable files pattern:

```bash
# Create environment-specific overrides
cat > vars/prod.yml <<EOF
password_max_days: 90
disable_ipv6: true
ssh_max_auth_tries: 3
EOF

ansible-playbook -i inventory/hosts.ini playbooks/site.yml \
  -e @vars/prod.yml
```

## Validation Tests

Run these from the control node (or via `ansible -m shell`) to verify hardening was applied:

### Filesystem — Unused modules blocked

```bash
ansible managed_nodes -i inventory/hosts.ini -m shell -a \
  "modprobe cramfs 2>&1 || echo 'PASS: cramfs blocked'"
# Expected: modprobe: ERROR: could not insert 'cramfs': Operation not permitted
```

### Network — sysctl values

```bash
ansible managed_nodes -i inventory/hosts.ini -m shell -a \
  "sysctl net.ipv4.ip_forward net.ipv4.conf.all.send_redirects net.ipv4.tcp_syncookies"
# Expected:
#   net.ipv4.ip_forward = 0
#   net.ipv4.conf.all.send_redirects = 0
#   net.ipv4.tcp_syncookies = 1
```

### SSH — daemon configuration

```bash
ansible managed_nodes -i inventory/hosts.ini -m shell -a \
  "sshd -T | grep -E 'permitrootlogin|passwordauthentication|x11forwarding|maxauthtries'"
# Expected:
#   permitrootlogin no
#   passwordauthentication no
#   x11forwarding no
#   maxauthtries 4
```

### auditd — service active

```bash
ansible managed_nodes -i inventory/hosts.ini -m shell -a \
  "systemctl is-active auditd"
# Expected: active
```

### rsyslog — service active

```bash
ansible managed_nodes -i inventory/hosts.ini -m shell -a \
  "systemctl is-active rsyslog"
# Expected: active
```

### Login banner

```bash
ansible managed_nodes -i inventory/hosts.ini -m shell -a \
  "cat /etc/issue"
# Expected: the AUTHORIZED ACCESS ONLY banner text
```

### ansible-lint — zero warnings

```bash
ansible-lint playbooks/site.yml
# Expected: no output (exit 0)
```

### Idempotency — second run shows zero changes

```bash
ansible-playbook -i inventory/hosts.ini playbooks/site.yml
# Expected PLAY RECAP: changed=0 for all hosts
```

## Teardown

```bash
cd terraform
terraform destroy \
  -var="public_key_path=~/.ssh/cis_hardening.pub" \
  -var="private_key_path=~/.ssh/cis_hardening" \
  -auto-approve
```

This destroys all AWS resources. The generated `inventory/hosts.ini` will remain locally but can be deleted manually.
